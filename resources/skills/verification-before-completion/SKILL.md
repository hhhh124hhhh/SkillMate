# 完成前验证

## 概述

在没有验证的情况下声称工作完成是不诚实，不是效率。

**核心原则**：证据优先于声明，始终。

**违反此规则的字面意思就是违反此规则的精神。**

## 铁律

```
没有新鲜验证证据就没有完成声明
```

如果你没有在此消息中运行验证命令，你就不能声称它通过。

## 门控功能

```
在声称任何状态或表达满足之前：

1. 识别：什么命令证明此声明？
2. 运行：执行完整命令（新鲜、完整）
3. 阅读：完整输出，检查退出代码，计数失败
4. 验证：输出确认声明吗？
   - 如果否：说明实际状态并附证据
   - 如果是：附证据说明声明
5. 只有这样：做出声明

跳过任何步骤 = 不诚实，不是验证
```

## 常见失败

| 声明 | 需要 | 不够 |
|------|------|------|
| 测试通过 | 测试命令输出：0 失败 | 之前运行，"应该通过" |
| Linter 干净 | Linter 输出：0 错误 | 部分检查，推断 |
| 构建成功 | 构建命令：退出 0 | Linter 通过，日志看起来不错 |
| Bug 修复 | 测试原始症状：通过 | 代码更改，假设修复 |
| 回归测试工作 | Red-green 循环验证 | 测试通过一次 |
| Agent 完成 | VCS diff 显示更改 | Agent 报告"成功" |
| 满足要求 | 逐行检查清单 | 测试通过 |

## 红旗 - 停止

- 使用"应该"、"可能"、"似乎"
- 在验证之前表达满意（"很好！"，"完美！"，"完成！"等）
- 即将提交/推送/PR 而没有验证
- 信任 agent 成功报告
- 依赖部分验证
- 想"就这一次"
- 累了并希望工作结束
- **任何暗示成功而没有运行验证的措辞**

## 合理化预防

| 借口 | 现实 |
|------|------|
| "现在应该工作了" | 运行验证 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter 通过了" | Linter ≠ 编译器 |
| "Agent 说成功" | 独立验证 |
| "我累了" | 疲劳 ≠ 借口 |
| "部分检查足够" | 部分证明不了什么 |
| "不同词语所以规则不适用" | 精神高于字母 |

## 关键模式

**测试**：
```
✅ [运行测试命令] [见：34/34 通过] "所有测试通过"
❌ "现在应该通过"/"看起来正确"
```

**回归测试（TDD Red-Green）**：
```
✅ 编写 → 运行（通过）→ 恢复修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我写了回归测试"（没有 red-green 验证）
```

**构建**：
```
✅ [运行构建] [见：退出 0] "构建通过"
❌ "Linter 通过"（linter 不检查编译）
```

**要求**：
```
✅ 重读计划 → 创建检查清单 → 验证每个 → 报告差距或完成
❌ "测试通过，阶段完成"
```

**Agent 委托**：
```
✅ Agent 报告成功 → 检查 VCS diff → 验证更改 → 报告实际状态
❌ 信任 agent 报告
```

## 为什么这很重要

从 24 次失败记忆：
- 人类伙伴说"我不相信你" - 信任破裂
- 未定义的函数已发布 - 会崩溃
- 缺失要求已发布 - 不完整的功能
- 浪费时间在虚假完成 → 重定向 → 返工
- 违反："诚实是核心价值观。如果你撒谎，你将被替换。"

## 何时应用

**总是在以下情况之前**：
- 任何成功/完成声明的变体
- 任何满意的表达
- 任何关于工作状态的积极声明
- 提交、PR 创建、任务完成
- 移动到下一个任务
- 委托给 agents

**规则适用于**：
- 确切短语
- 解释和同义词
- 成功的暗示
- 任何建议完成/正确的沟通

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后声明结果。

这是不可协商的。
